# 実装計画

- [x] 1. DB 更新の一時的な失敗を吸収できる再試行の仕組みを追加する
- [x] 1.1 再試行方針を定義する
  - 一時的な失敗時に、一定回数まで待機を挟んで再試行できるようにする
  - 呼び出し元が共通的に利用できるよう、DB 更新処理を安全に包める形にする
  - 再試行回数や待機方針が一貫して適用されるようにする
  - _Requirements: 1.4, 1.5_

- [x] 1.2 再試行時の振る舞いを実装する
  - 各試行で DB 更新を実行し、成功した時点で処理を完了する
  - 失敗時は次の試行まで待機し、上限に達するまで継続する
  - 再試行で回復しない場合は失敗を呼び出し元へ返す
  - _Requirements: 1.1, 1.2, 1.3, 1.5_

- [x] 1.3 失敗状況を追跡できるログ出力を追加する
  - 再試行途中の失敗について、対象・試行回数・エラー内容が分かる形で記録する
  - 再試行を使い切っても回復しなかった場合に、恒久的な失敗として識別できる形で記録する
  - _Requirements: 2.1, 2.2, 2.3_

- [x] 2. Issue をクローズする処理で再試行の仕組みを利用する
  - 外部操作が成功した後の DB 反映が、一時的失敗では直ちに失われないようにする
  - メモリ上の状態更新は、永続化の成功後にのみ反映されることを維持する
  - _Requirements: 1.1_

- [x] 3. ワークツリーのクリーンアップ処理で再試行の仕組みを利用する
  - 後続の DB 反映が一時的失敗で欠落しないようにする
  - メモリ上の状態更新は、永続化の成功後にのみ反映されることを確認する
  - _Requirements: 1.2_

- [x] 4. 再試行の仕組みを検証するテストを追加する
- [x] 4.1 (P) 一時的失敗から回復するケースを検証する
  - 最初は失敗し、その後成功する更新処理を用意する
  - 最終的に成功として扱われることを検証する
  - 想定どおり再試行が行われることを検証する
  - _Requirements: 3.1, 3.3_

- [x] 4.2 (P) 回復不能な失敗のケースを検証する
  - すべての試行で失敗する更新処理を用意する
  - 最終的に失敗として返されることを検証する
  - 上限回数まで再試行されることを検証する
  - _Requirements: 3.2, 3.3, 3.4_
