# リサーチ・設計判断ログ

---
**目的**: 設計フェーズにおけるディスカバリー結果、アーキテクチャ調査、設計判断の根拠を記録する。

---

## サマリー

- **フィーチャー**: `adr-setup`
- **ディスカバリースコープ**: Simple Addition（ドキュメント追加）
- **主な知見**:
  - 本フィーチャーはコードではなくドキュメントの追加であるため、既存アーキテクチャへの影響は軽微
  - AGENTS.md・CLAUDE.md は単純なテキストファイルであり、既存構造を壊さず追記すれば足りる
  - ADR の標準フォーマット（Michael Nygard 式）を採用することで将来の参照性が高まる

## リサーチログ

### ADR フォーマット標準

- **コンテキスト**: 独自フォーマットを策定するか、既存標準を採用するかの検討
- **参照資料**: Michael Nygard の ADR テンプレート（業界標準）、Issue の ADR フォーマット案
- **知見**:
  - Issue 記載のフォーマット（ステータス / コンテキスト / 決定 / 理由 / 却下した代替案）は Nygard 式の簡略版として広く普及している
  - ステータス管理（Accepted / Superseded / Deprecated）は将来の更新追跡に必要
  - ファイル名に連番プレフィックスを付けること（`001-`, `002-`）で時系列と検索性を両立できる
- **示唆**: Issue のフォーマット案をそのまま採用する。追加フィールドは不要

### 既存ファイル構造の調査

- **コンテキスト**: AGENTS.md と CLAUDE.md に追記する位置の確認
- **調査結果**:
  - `AGENTS.md` は現在「Quality Check」のみを含むシンプルな構造
  - `CLAUDE.md` はプロジェクト開発ガイドライン（Paths / Steering / Workflow）を含む
- **示唆**:
  - `AGENTS.md` には「設計判断の参照先」として新セクションを追加するか、既存セクション末尾に追記する
  - `CLAUDE.md` の「Project Context > Paths」セクションに ADR パスを追記するのが自然

## アーキテクチャパターン評価

| オプション | 説明 | 強み | リスク | 備考 |
|------------|------|------|--------|------|
| `docs/adr/` 配置 | ドキュメント専用ディレクトリ配下に集約 | 検索しやすい、慣例に沿っている | なし | GitHub での標準的配置 |
| `adr/` ルート配置 | リポジトリルート直下 | シンプル | `docs/` との不整合 | 本プロジェクトには `docs/` が存在しないため要作成 |

→ `docs/adr/` を採用（Issue の指定通り、かつ業界慣行に合致）

## 設計判断

### 判断: AGENTS.md への追記位置

- **コンテキスト**: AGENTS.md は現在「Quality Check」のみの短いファイル
- **検討した選択肢**:
  1. 既存セクション末尾への追記
  2. 新セクション「Design References」を追加
- **選択**: 新セクション「Design References」を追加し、ADR への参照リンクを配置
- **理由**: Quality Check とは異なる用途（設計判断参照）であるため、セクションを分けた方が明確
- **トレードオフ**: ファイルが若干長くなるが、AIエージェントの参照精度が向上する

### 判断: CLAUDE.md への追記位置

- **コンテキスト**: CLAUDE.md の「Project Context > Paths」セクションに Steering / Specs のパスが既に記載されている
- **選択**: 同セクションに `ADR: docs/adr/` を追記
- **理由**: Paths セクションは参照先一覧を集約する場所であり、ADR の追記が最も自然

## リスクと緩和策

- `.gitkeep` を置かない場合、空ディレクトリが git にコミットされない → `template.md` が `docs/adr/` の最初のファイルとなるため問題なし
- ADR 番号の重複 → 連番 `NNN-` プレフィックスで管理し、新規 ADR は次番から採番する規則をテンプレートに明記する
