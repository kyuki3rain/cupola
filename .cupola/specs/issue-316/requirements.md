# 要件定義書

## はじめに

`PostCiFixLimitComment` エフェクトは best-effort として実行されるが、投稿失敗時でも `ci_fix_count` がインクリメントされ永続化されるため、次のポーリングサイクルで再発火条件が成立せず、ユーザーへの通知が永久に欠落するリスクがある（`docs/architecture/effects.md:56` にて既知リスクとして明記済み）。

本仕様は最小スコープで以下を実装する。

1. Issue に `ci_fix_limit_notified` フラグを追加し、Decide が「条件成立かつ未通知」の場合のみ発火、Execute の成功時のみフラグを書き込む仕組みを実装する
2. `status` / `doctor` コマンドに「未送信通知あり」の警告を追加し、運用者が状況を把握できる動線を確保する

長期的な汎用解（EffectLog port + 観測の完全抽象化）は #338 で別途対応する。

## 要件

### 要件 1: Issue ドメインモデルへの通知済みフラグの追加

**目的:** 開発者として、Issue に CI 修正上限通知の送信状態を表す `ci_fix_limit_notified` フラグを追加したい。これにより、通知失敗時に次サイクルで Decide が再発火できるようになる。

#### 受け入れ条件
1. The system shall expose a `ci_fix_limit_notified: bool` field on the `Issue` struct.
2. When a new `Issue` is constructed via `Issue::new`, the system shall initialize `ci_fix_limit_notified` to `false`.
3. The system shall include `ci_fix_limit_notified: Option<bool>` in `MetadataUpdates` to support sparse updates.
4. When `MetadataUpdates.ci_fix_limit_notified` is `Some(v)`, the `apply_to` method shall update `issue.ci_fix_limit_notified` to `v`.

### 要件 2: DB スキーマ・マイグレーション

**目的:** オペレーターとして、既存の本番データベースを壊さずに `ci_fix_limit_notified` 列を追加したい。これにより、既存の Cupola 運用環境がスキーママイグレーションを経て継続動作できる。

#### 受け入れ条件
1. The system shall define `ci_fix_limit_notified INTEGER NOT NULL DEFAULT 0` in the `CREATE TABLE issues` statement.
2. When `init_schema` is called on an existing database without the `ci_fix_limit_notified` column, the system shall execute an `ALTER TABLE issues ADD COLUMN` migration without affecting existing rows.
3. The system shall map the `ci_fix_limit_notified` column value (`0`/`1`) to Rust `bool` in all `SELECT` queries via `row_to_issue`.
4. The system shall persist `ci_fix_limit_notified` in `INSERT` statements for new issues.
5. When `MetadataUpdates.ci_fix_limit_notified` is `Some`, the system shall include `ci_fix_limit_notified` in the dynamic `UPDATE` SQL generated by `update_state_and_metadata`.

### 要件 3: Decide ロジックの発火条件変更

**目的:** システムとして、`PostCiFixLimitComment` を「CI 修正上限超過かつ未通知」のときのみ発火させたい。これにより、通知失敗後の次サイクルで再発火が保証される。

#### 受け入れ条件
1. While `prev.ci_fix_count >= cfg.max_ci_fix_cycles` and `prev.ci_fix_limit_notified == false`, the system shall emit `Effect::PostCiFixLimitComment` and set `metadata_updates.ci_fix_count = Some(prev.ci_fix_count + 1)`.
2. While `prev.ci_fix_limit_notified == true`, the system shall not emit `Effect::PostCiFixLimitComment` regardless of `ci_fix_count`.
3. The system shall apply the updated condition to both the CI failure path (`decide.rs` 旧 line 681) and the conflict path (旧 line 693) in `decide_implementation_review_waiting`.
4. When emitting `PostCiFixLimitComment`, the system shall NOT set `ci_fix_limit_notified` in `MetadataUpdates`（execute が成功時のみ書き込む）。

### 要件 4: Execute での成功時フラグ書き込み

**目的:** システムとして、`PostCiFixLimitComment` エフェクトが成功した場合のみ `ci_fix_limit_notified = true` を永続化したい。これにより、失敗時に次のポーリングサイクルで再発火できる。

#### 受け入れ条件
1. When `Effect::PostCiFixLimitComment` is executed successfully, the system shall call `issue_repo.update_state_and_metadata(issue.id, MetadataUpdates { ci_fix_limit_notified: Some(true), ..Default::default() })`.
2. If `Effect::PostCiFixLimitComment` execution fails (GitHub API error), the system shall log a warning and shall not persist `ci_fix_limit_notified = true`, leaving it as `false` in the database.
3. The system shall maintain `Effect::PostCiFixLimitComment` as best-effort (non-fatal); the overall polling cycle shall continue after a posting failure.

### 要件 5: status コマンドへの警告表示

**目的:** オペレーターとして、`cupola status` で「CI 修正上限通知が未送信の Issue」を一目で把握したい。これにより、GitHub 通信障害時の通知欠落に素早く気付ける。

#### 受け入れ条件
1. When an Issue satisfies `ci_fix_count > cfg.max_ci_fix_cycles && !ci_fix_limit_notified`, the system shall display `⚠ ci-fix-limit notification pending` alongside that Issue's entry in `status` output.
2. While no Issue satisfies the pending-notification condition, the system shall not display the warning in `status` output.

### 要件 6: doctor コマンドへの警告チェック追加

**目的:** オペレーターとして、`cupola doctor` の `OperationalReadiness` セクションで「CI 修正上限通知が未送信の Issue 数」を確認したい。これにより、GitHub 通信障害の影響を定量的に把握し、remediation を行える。

#### 受け入れ条件
1. When one or more Issues satisfy `ci_fix_count > cfg.max_ci_fix_cycles && !ci_fix_limit_notified`, the system shall include a `CheckStatus::Warn` result in `OperationalReadiness` reporting the pending count.
2. When no Issue satisfies the pending-notification condition, the system shall include a `CheckStatus::Ok` result in `OperationalReadiness`.
3. The remediation message shall instruct the operator to check GitHub connectivity and re-run `cupola start` to resume polling.
4. The `DoctorUseCase` shall accept an `IssueRepository` port (as a generic type parameter) and `max_ci_fix_cycles: u32` to query the pending count, making the check testable with mock adapters.
