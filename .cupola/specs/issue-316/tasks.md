# 実装タスクリスト

## タスク概要

全体の依存関係:
1. **タスク 1**（ドメインモデル）を最初に完了させる
2. タスク 1 の完了後、**タスク 2・3・5** が並列実行可能
3. タスク 2 の完了後、**タスク 4・6** が並列実行可能
4. すべてのタスクの完了後、**タスク 7**（テスト強化）を実施

---

- [ ] 1. 通知済みフラグをドメインで扱えるようにする（ドメイン層）
- [ ] 1.1 Issue が CI 修正上限の通知済み状態を保持できるようにする
  - Issue の生成直後は、通知未済として一貫した初期状態になること
  - 既存のドメインテストが、新しい状態を含めても意味を保って成立すること
  - _Requirements: 1.1, 1.2_

- [ ] 1.2 メタデータ更新で通知済み状態を変更できるようにする
  - 通知済み状態を更新対象に含めるかどうかを表現できること
  - 更新対象に含まれる場合のみ、Issue の通知済み状態へ反映されること
  - 既存の更新系テストが、新しい状態を含めても成立すること
  - _Requirements: 1.3, 1.4_

---

- [ ] 2. 永続化レイヤーで通知済みフラグを保存・復元・更新できるようにする（タスク 1 完了後）
- [ ] 2.1 新規作成される Issue データに通知済み状態が保持されるようにする
  - 新規に初期化される保存領域で、通知済み状態を欠落なく保持できること
  - _Requirements: 2.1_

- [ ] 2.2 既存データでも通知済み状態を安全に扱えるようにする
  - 既存の保存領域に対しても、互換性を保ったまま初期化・移行が完了すること
  - 通知済み状態を持たない既存データがあっても、正常に扱えること
  - _Requirements: 2.2_

- [ ] 2.3 (P) 読み出し時に通知済み状態が復元されるようにする
  - Issue を取得する各経路で、通知済み状態が欠落せずドメインへ反映されること
  - _Requirements: 2.3_

- [ ] 2.4 (P) 保存時に通知済み状態が永続化されるようにする
  - 新規保存した Issue について、通知済み状態が再読み出し時に維持されること
  - _Requirements: 2.4_

- [ ] 2.5 (P) 部分更新で通知済み状態を変更できるようにする
  - 通知済み状態を更新対象に含めた場合のみ、既存データへ反映されること
  - 他の更新項目と組み合わせても、従来の更新動作を壊さないこと
  - _Requirements: 2.5_

---

- [ ] 3. 発火条件を「上限超過かつ未通知」に変更する（タスク 1 完了後、タスク 2 と並列可）
- [ ] 3.1 (P) CI failure 経路で「上限超過かつ未通知」の場合のみ PostCiFixLimitComment が発火するようにする
  - ci_fix_count が上限以上かつ通知未済の Issue に対して、コメント通知の Effect が生成されること
  - 通知済みの Issue に対しては Effect が生成されないこと
  - _Requirements: 3.1, 3.2, 3.4_

- [ ] 3.2 (P) conflict 経路でも同様に「上限超過かつ未通知」の場合のみ PostCiFixLimitComment が発火するようにする
  - conflict 経路においても CI failure 経路と同一の発火条件が適用されること
  - _Requirements: 3.1, 3.2, 3.3, 3.4_

---

- [ ] 4. 通知成功時のみ通知済み状態を永続化する（タスク 2 完了後）
- [ ] 4.1 PostCiFixLimitComment の投稿成否に応じて通知済み状態を管理する
  - GitHub への投稿が成功した場合のみ、通知済みフラグが DB に書き込まれること
  - GitHub への投稿が失敗した場合は、通知済みフラグが変更されないこと（次サイクルで再発火可能）
  - 投稿失敗は呼び出し元に伝播せず、警告ログが残ること
  - _Requirements: 4.1, 4.2, 4.3_

---

- [ ] 5. status コマンドで通知待ち状態の Issue を識別できるようにする（タスク 1 完了後、タスク 3/4 と並列可）
- [ ] 5.1 (P) CI 修正上限を超過しているが未通知の Issue に警告マークが表示されること
  - 通知待ち状態（上限超過かつ未通知）の Issue には、status 出力で視覚的な警告が付加されること
  - 通知待ちでない Issue の表示は変わらないこと
  - _Requirements: 5.1, 5.2_

---

- [ ] 6. doctor コマンドで通知待ち件数を診断できるようにする（タスク 2 完了後）
- [ ] 6.1 DoctorUseCase が Issue の永続化データを参照できるようになること
  - DoctorUseCase が IssueRepository を通じて Issue 一覧を取得できること
  - DoctorUseCase::run が非同期で呼び出せること
  - _Requirements: 6.4_

- [ ] 6.2 通知待ち Issue が存在する場合に doctor が警告を出力すること
  - 上限超過かつ未通知の Issue が 1 件以上あれば、OperationalReadiness セクションで Warn が返ること
  - 通知待ちが 0 件の場合は Ok が返ること
  - 警告に対して remediation メッセージが付与されること
  - _Requirements: 6.1, 6.2, 6.3_

---

- [ ] 7. テスト強化
- [ ] 7.1 Decide の再発火・抑制ロジックをユニットテストで保証する
  - count が上限以上かつ未通知の場合に PostCiFixLimitComment が発火すること（CI failure 経路・conflict 経路の両方）
  - 通知済みの場合は発火しないこと
  - count が上限未満の場合は発火しないこと
  - _Requirements: 3.1, 3.2_

- [ ] 7.2 (P) Execute での成功時フラグ書き込み・失敗時スキップをユニットテストで保証する
  - GitHub 投稿成功時に通知済みフラグの永続化が呼ばれること
  - GitHub 投稿失敗時に通知済みフラグの永続化が呼ばれないこと
  - _Requirements: 4.1, 4.2_

- [ ] 7.3 (P) 永続化のラウンドトリップを SQLite 統合テストで保証する
  - 新規スキーマに通知済み状態の列が含まれること
  - 既存スキーマへのマイグレーションが安全に完了すること
  - 保存・取得のラウンドトリップで通知済み状態が正しく保持されること
  - 部分更新で通知済み状態を変更できること
  - _Requirements: 2.1, 2.2, 2.3, 2.4, 2.5_

- [ ] 7.4 (P) DoctorUseCase の警告・正常判定をユニットテストで保証する
  - 通知待ち件数に応じて Warn / Ok が返ること
  - 返却値が OperationalReadiness セクションに属すること
  - 警告時の remediation メッセージが適切であること
  - _Requirements: 6.1, 6.2, 6.3_
