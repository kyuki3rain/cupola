# 実装計画

- [ ] 1. secrecy を利用できるようにする
  - Unix 向け依存関係に `secrecy` を追加し、秘密情報型を利用できる状態にする
  - コンパイル確認を行い、依存追加によるビルド破壊がないことを確認する
  - _Requirements: 1.1, 1.2_

- [ ] 2. アダプター層でトークンを秘密情報型として扱えるようにする

- [ ] 2.1 (P) REST クライアントでのトークン保持と認証ヘッダ生成を更新する
  - REST クライアントが秘密情報型のトークンを受け取って保持できるようにする
  - 認証に利用する箇所では、秘密値を明示的に取り出してヘッダや外部ライブラリ呼び出しへ渡すようにする
  - テスト用の初期化経路でも本番と同じ秘密情報型を利用する
  - _Requirements: 2.1, 2.3, 2.5_

- [ ] 2.2 (P) GraphQL クライアントのトークン取り扱いを SecretString ベースに更新する
  - GraphQL クライアントが保持する認証トークンを `SecretString` として扱う
  - GraphQL クライアントの初期化経路でも `SecretString` を受け取る前提にそろえる
  - 認証ヘッダー等でトークン文字列が必要な箇所では、`SecretString` からの取り出しが明示的に行われるようにする
  - これにより、通常の保持・受け渡しでは平文文字列を避けつつ、必要時のみトークン文字列へアクセスする
  - _Requirements: 2.2, 2.4, 2.5_

- [ ] 3. bootstrap でのトークン受け渡しを更新する
  - 起動時に取得した生トークンを `SecretString` へラップして扱う
  - ラップ済みのトークンが各起動経路から利用される GitHub クライアントへ一貫して渡されるようにする
  - トークンの型変更後も既存の起動フローとクライアント生成の挙動を維持する
  - _Requirements: 3.1, 3.2_

- [ ] 4. Debug マスキングのテストを追加し品質確認する

- [ ] 4.1 秘密情報型の Debug マスキングを検証するテストを追加する
  - 秘密情報型のデバッグ出力に実トークン文字列が含まれないことを確認するテストを追加する
  - マスキングによりログや失敗時出力から秘密情報が露出しないことを担保する
  - _Requirements: 4.1, 4.2_

- [ ] 4.2 品質保証を確認する
  - lint を実行し、警告を含めて品質ゲートを通過することを確認する
  - 全テストを実行し、既存のページネーション関連テストを含めて成功することを確認する
  - _Requirements: 5.1, 5.2, 5.3_
