# Implementation Plan

- [x] 1. Link ヘッダーから次ページ URL を抽出できるようにする
- [x] 1.1 RFC 5988 形式の Link ヘッダーを解析し、指定した rel の URL を取得する機能を実装する
  - `rel="next"` のような rel 属性を指定して対応する URL を取得できる
  - 指定した rel が存在しない場合は「なし」を返す
  - _Requirements: 1.3_

- [x] 1.2 (P) Link ヘッダーパース機能のユニットテストを追加する
  - `rel="next"` を含む場合に URL が正しく取得できることを検証する
  - `rel="next"` が存在しない場合に「なし」が返ることを検証する
  - `rel="next"` と `rel="last"` が混在する場合に対象の rel のみ取得できることを検証する
  - _Requirements: 3.4_

- [x] 2. ページネーション上限値を管理できるようにする
  - 無限ループ防止のためのページ数上限をコード中に名前付き定数として定義する
  - _Requirements: 2.3_

- [x] 3. ラベル付与 actor 取得を全ページ対応に拡張する
- [x] 3.1 レスポンスヘッダーを body 消費前に安全に取得できるよう処理順序を変更する
  - Link ヘッダーの取得を body の読み取りよりも先に行うことで Rust の所有権制約を回避する
  - _Requirements: 1.4_

- [x] 3.2 全ページのイベントを収集してから最新の labeled イベントを検索できるようにする
  - 次ページが存在する間はリクエストを繰り返し、全ページのイベントを結合する
  - 次ページが存在しない場合、またはページ上限に達した場合にループを終了する
  - 収集した全イベントを逆順で走査し、最新のラベル付与 actor を返す
  - _Requirements: 1.1, 1.2, 1.3, 2.1, 2.2_

- [x] 4. ページネーション統合テスト用の mock HTTP サーバーを導入する
  - テストで使用する mock HTTP サーバークレートを選定し dev-dependency に追加する（`wiremock` や `mockito` など、既存依存関係で代替できない場合は追加する）
  - _Requirements: 3.1, 3.2, 3.3, 3.5_

- [x] 5. ページネーションの統合テストを追加する
- [x] 5.1 (P) 複数ページにまたがるイベントをすべて取得できることを検証する
  - 1 ページ目に「次のページあり」を示す Link ヘッダーを含む応答を返し、2 ページ目には含まれないよう設定する
  - 両ページのイベントが結合されて取得されることを確認する
  - _Requirements: 3.1_

- [x] 5.2 (P) 後続ページに存在するラベル付与イベントから actor を特定できることを検証する
  - 対象ラベルの付与イベントが最初のページではなく後続ページに存在する状況を再現する
  - そのイベントから actor のログインが正しく返されることを確認する
  - _Requirements: 3.2_

- [x] 5.3 (P) 次ページが存在しない場合にループが終了することを検証する
  - Link ヘッダーを含まない単一ページの応答を返す状況を設定する
  - 追加のリクエストなしに処理が完了することを確認する
  - _Requirements: 3.3_

- [x] 5.4 (P) ページ上限に達した場合に超過リクエストが送信されないことを検証する
  - 上限を超えるページ数の応答を返せる状況を設定する
  - 上限ページ分のリクエストのみが実行され、それ以上のリクエストが送信されないことを確認する
  - _Requirements: 3.5_
