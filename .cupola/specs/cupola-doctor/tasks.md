# Implementation Plan

- [x] 1. Domain 層のチェック結果モデルを定義する
  - チェック結果のステータスを表す列挙型（成功・失敗・スキップ）を作成する
  - チェック名、ステータス、修正手順を保持する構造体を作成する
  - チェック結果表示に必要な生データ（チェック名、ステータス、修正手順メッセージなど）だけを持たせる（✅/❌ やメッセージの整形は CLI / presenter 側で行う）
  - _Requirements: 2.4, 3.2, 6.1, 6.2, 6.3_

- [x] 2. 外部コマンド実行の抽象化を構築する
- [x] 2.1 (P) コマンド実行ポートを定義する
  - コマンド名と引数を受け取り、成功/失敗・標準出力・標準エラー出力を返すインターフェースを定義する
  - コマンド実行結果を表すデータ構造を作成する
  - _Requirements: 2.1, 2.2, 2.3, 3.1_

- [x] 2.2 (P) コマンド実行アダプターを実装する
  - OS のプロセス実行機能を使用してコマンドを実行する
  - コマンドが見つからない場合は失敗として処理する（パニックしない）
  - _Requirements: 2.1, 2.2, 2.3, 3.1_

- [x] 3. doctor ユースケースを実装する
- [x] 3.1 外部ツールチェック機能を実装する
  - git のバージョン確認コマンドを実行し、利用可能かチェックする
  - Claude Code CLI のバージョン確認コマンドを実行し、利用可能かチェックする
  - GitHub CLI の認証状態を確認し、認証済みかチェックする
  - 各チェックの失敗時に対応する修正手順メッセージを設定する
  - _Requirements: 2.1, 2.2, 2.3, 2.4_

- [x] 3.2 agent:ready ラベルチェック機能を実装する
  - GitHub CLI 認証チェックの結果を確認し、失敗時はラベルチェックをスキップする
  - 認証成功時は、ラベル一覧を JSON 形式で取得し `agent:ready` ラベルの存在を確認する
  - スキップ時はスキップ状態のチェック結果を返す（失敗としてカウントしない）
  - _Requirements: 3.1, 3.2_

- [x] 3.3 設定ファイルとプロジェクト構成チェック機能を実装する
  - `cupola.toml` の存在確認と必須項目（owner, repo, default_branch）の検証を行う（既存の設定ローダーを再利用）
  - steering ディレクトリにファイルが 1 つ以上存在するか確認する
  - データベースファイルの存在を確認する（初期化済みかの判定）
  - 各チェックの失敗時に対応する修正手順メッセージを設定する
  - _Requirements: 4.1, 4.2, 4.3, 5.1, 5.2, 5.3_

- [x] 3.4 チェック実行のオーケストレーションを実装する
  - 全チェック項目を定められた順序（git → claude → gh → label → toml → steering → db）で実行する
  - 全チェック項目の結果リストを返す
  - _Requirements: 1.1_

- [x] 4. CLI サブコマンドの登録と統合を行う
  - CLI のサブコマンド定義に doctor を追加する
  - ブートストラップ層のコマンドディスパッチに doctor の分岐を追加する
  - ユースケースのインスタンス化と依存注入を行う
  - チェック結果をフォーマットして標準出力に表示する（成功時は ✅、失敗時は ❌ とチェック名を表示し、失敗時は修正手順も表示する）
  - チェック結果に基づき終了コードを設定する（全成功: 0、失敗あり: 1）
  - _Requirements: 1.1, 1.2, 6.1, 6.2, 6.3, 7.1, 7.2_

- [x] 5. テストを実装する
- [x] 5.1 (P) ユニットテストを実装する
  - モックのコマンド実行を注入し、全チェック成功パターンをテストする
  - 外部ツールチェック失敗時の結果と修正手順を検証する
  - gh 認証失敗時にラベルチェックがスキップされることを検証する
  - 設定ファイル不在時やディレクトリ空時のチェック失敗を検証する
  - _Requirements: 2.1, 2.2, 2.3, 2.4, 3.1, 3.2, 4.1, 4.2, 4.3, 5.1, 5.2, 5.3, 6.1, 6.2, 6.3_

- [x] 5.2 (P) 統合テストを実装する
  - コマンド実行アダプターの実際のコマンド実行を検証する（git --version など）
  - tempdir を使用してファイルシステムチェック（toml、steering、db）の動作を検証する
  - 全チェック成功時の exit code 0 と失敗時の exit code 1 を検証する
  - _Requirements: 7.1, 7.2_
